More specific than a Pillar Weakness, but more general than a Base Weakness. Class level weaknesses typically describe issues in terms of 1 or 2 of the following dimensions: behavior, property, and resource.</div><p>View customized information:</p>For users who are interested in more notional aspects of a weakness. Example: educators, technical writers, and project/program managers. For users who are concerned with the practical application and details about the nature of a weakness and how to prevent it from happening. Example: tool developers, security researchers, pen-testers, incident response analysts. For users who are mapping an issue to CWE/CAPEC IDs, i.e., finding the most appropriate CWE for a specific issue (e.g., a CVE record). Example: tool developers, security researchers. For users who wish to see all available information for the CWE/CAPEC entry. For users who want to customize what details are displayed.<p>Description</p><div><table><tbody><tr><td>The product does not properly assign, modify, track, or check privileges for an actor, creating an unintended sphere of control for that actor.</td><td><a href="https://cwe.mitre.org/data/images/CWE-269-Diagram.png"><img src="https://cwe.mitre.org/data/images/CWE-269-Diagram.png"></a></td></tr></tbody></table></div><p>Common Consequences</p><div>This table specifies different individual consequences associated with the weakness. The Scope identifies the application security area that is violated, while the Impact describes the negative technical impact that arises if an adversary succeeds in exploiting this weakness. The Likelihood provides information about how likely the specific consequence is expected to be seen relative to the other consequences in the list. For example, there may be high likelihood that a weakness will be exploited to achieve a certain impact, but a low likelihood that it will be exploited to achieve a different impact.<table width="98%"><tbody><tr><th width="135px">Impact</th><th>Details</th></tr><tr><td>Scope: Access Control</td></tr></tbody></table></div><p>Potential Mitigations</p><div><table width="98%"><tbody><tr><th>Phase(s)</th><th>Mitigation</th></tr><tr><td></td><td><p>Very carefully manage the setting, management, and handling of privileges. Explicitly manage trust zones in the software.</p></td></tr><tr><td></td><td><p>Strategy: <i>Separation of Privilege</i></p><p>Follow the principle of least privilege when assigning access rights to entities in a software system.</p></td></tr><tr><td></td><td><p>Strategy: <i>Separation of Privilege</i></p><p>Consider following the principle of separation of privilege. Require multiple conditions to be met before permitting access to a system resource.</p></td></tr></tbody></table></div><p>Relationships</p><p>This table shows the weaknesses and high level categories that are related to this weakness. These relationships are defined as ChildOf, ParentOf, MemberOf and give insight to similar items that may exist at higher and lower levels of abstraction. In addition, relationships such as PeerOf and CanAlsoBe are defined to show similar weaknesses that the user may want to explore.</p><p>Modes Of Introduction</p><div>The different Modes of Introduction provide information about how and when this weakness may be introduced. The Phase identifies a point in the life cycle at which introduction may occur, while the Note provides a typical scenario related to introduction during the given phase.<table width="98%"><tbody><tr><th width="110px">Phase</th><th>Note</th></tr><tr><td width="100%"></td></tr><tr><td width="100%">REALIZATION: This weakness is caused during implementation of an architectural security tactic.</td></tr><tr><td width="100%"></td></tr></tbody></table></div><p>Applicable Platforms</p>This listing shows possible areas for which the given weakness could appear. These may be for specific named Languages, Operating Systems, Architectures, Paradigms, Technologies, or a class of such platforms. The platform is listed along with how frequently the given weakness appears for that instance.<p>Class: Not Language-Specific (Undetermined Prevalence)</p><p>Likelihood Of Exploit</p><p>Medium</p><p>Demonstrative Examples</p><div><br><p>This code temporarily raises the program's privileges to allow creation of a new user folder.</p><p>(bad code)</p><p>Example Language: Python</p><div>if invalidUsername(username):<p><br><i>#avoid <a href="https://cwe.mitre.org/data/definitions/22.html">CWE-22</a> and <a href="https://cwe.mitre.org/data/definitions/78.html">CWE-78</a></i><br>print('Usernames cannot contain invalid characters')<br>return False</p><br>try:<p>raisePrivileges()<br>os.mkdir('/home/' + username)<br>lowerPrivileges()</p><br>except OSError:<p>print('Unable to create new user directory for user:' + username)<br>return False</p><br>return True</div><p>While the program only raises its privilege level to create the folder and immediately lowers it again, if the call to os.mkdir() throws an exception, the call to lowerPrivileges() will not occur. As a result, the program is indefinitely operating in a raised privilege state, possibly allowing further exploitation to occur.</p><br><br><p>The following example demonstrates the weakness.</p><p>(bad code)</p><p>Example Language: C</p><p>seteuid(0);<br><i>/* do some stuff */</i><br><br>seteuid(getuid());</p><br><br><p>The following example demonstrates the weakness.</p><p>(bad code)</p><p>Example Language: Java</p>public Object run() {<p><i>// privileged code goes here, for example:</i><br>System.loadLibrary("awt");<br>return null;<br><i>// nothing to return</i><br></p>}<br><br><p>This code intends to allow only Administrators to print debug information about a system.</p><p>(bad code)</p><p>Example Language: Java</p><div>public enum Roles {<p>ADMIN,USER,GUEST</p>}<br><br>public void printDebugInfo(User requestingUser){<div>if(isAuthenticated(requestingUser)){<div>case GUEST:<p>System.out.println("You are not authorized to perform this command");<br>break;</p><br>default:<p>System.out.println(currentDebugState());<br>break;</p></div>}<br>else{<p>System.out.println("You must be logged in to perform this command");</p>}</div>}</div><p>While the intention was to only allow Administrators to print the debug information, the code as written only excludes those with the role of "GUEST". Someone with the role of "ADMIN" or "USER" will be allowed access, which goes against the original intent. An attacker may be able to use this debug information to craft an attack on the system.</p><br><br><p>This code allows someone with the role of "ADMIN" or "OPERATOR" to reset a user's password. The role of "OPERATOR" is intended to have less privileges than an "ADMIN", but still be able to help users with small issues such as forgotten passwords.</p><p>(bad code)</p><p>Example Language: Java</p><div>public enum Roles {<p>ADMIN,OPERATOR,USER,GUEST</p>}<br><br>public void resetPassword(User requestingUser, User user, String password ){<div>if(isAuthenticated(requestingUser)){<div>case GUEST:<p>System.out.println("You are not authorized to perform this command");<br>break;</p><br>case USER:<p>System.out.println("You are not authorized to perform this command");<br>break;</p><br>default:<p>setPassword(user,password);<br>break;</p>}</div><br>else{<p>System.out.println("You must be logged in to perform this command");</p>}</div>}</div><p>This code does not check the role of the user whose password is being reset. It is possible for an Operator to gain Admin privileges by resetting the password of an Admin account and taking control of that account.</p><br><br></div><p>Selected Observed Examples</p><p><i>Note: this is a curated list of examples for users to understand the variety of ways in which this weakness can be introduced.

## Conventions

The textual format for WebAssembly [modules](https://webassembly.github.io/spec/core/intro/overview.html#module) is a rendering of their [abstract syntax](https://webassembly.github.io/spec/core/syntax/modules.html#syntax-module) into [S-expressions](https://en.wikipedia.org/wiki/S-expression).

Like the [binary format](https://webassembly.github.io/spec/core/binary/index.html#binary), the text format is defined by an *attribute grammar*. A text string is a well-formed description of a module if and only if it is generated by the grammar. Each production of this grammar has at most one synthesized attribute: the abstract syntax that the respective character sequence expresses. Thus, the attribute grammar implicitly defines a *parsing* function. Some productions also take a as an inherited attribute that records bound [identifiers](https://webassembly.github.io/spec/core/text/values.html#text-id).

Except for a few exceptions, the core of the text grammar closely mirrors the grammar of the abstract syntax. However, it also defines a number of *abbreviations* that are “syntactic sugar” over the core syntax.

The recommended extension for files containing WebAssembly modules in text format is “`.wat` ”. Files with this extension are assumed to be encoded in UTF-8, as per [Unicode](https://www.unicode.org/versions/latest/) (Section 2.5).

## Contexts

The text format allows the use of symbolic [identifiers](https://webassembly.github.io/spec/core/text/values.html#text-id) in place of [indices](https://webassembly.github.io/spec/core/syntax/modules.html#syntax-index). To resolve these identifiers into concrete indices, some grammar productions are indexed by an *identifier context* as a synthesized attribute that records the declared identifiers in each [index space](https://webassembly.github.io/spec/core/syntax/modules.html#syntax-index). In addition, the context records the types defined in the module, so that [parameter](https://webassembly.github.io/spec/core/text/types.html#text-param) indices can be computed for [functions](https://webassembly.github.io/spec/core/text/modules.html#text-func).

For each index space, such a context contains the list of [names](https://webassembly.github.io/spec/core/syntax/values.html#syntax-name) assigned to the defined indices, which were denoted by the corresponding [identifiers](https://webassembly.github.io/spec/core/text/values.html#text-id). Unnamed indices are associated with empty () entries in these lists. Fields have *dependent* name spaces, and hence a separate list of field identifiers per type.

In addition, the field records the [defined type](https://webassembly.github.io/spec/core/valid/conventions.html#syntax-deftype) associated with each [type index](https://webassembly.github.io/spec/core/syntax/modules.html#syntax-typeidx). They are needed to look up the number of parameters of [function types](https://webassembly.github.io/spec/core/syntax/types.html#syntax-functype) when used in a [function definition](https://webassembly.github.io/spec/core/text/modules.html#text-func), in order to produce the correct indices for [locals](https://webassembly.github.io/spec/core/syntax/modules.html#syntax-local).

An identifier context is *well-formed* if no index space contains duplicate identifiers. For fields, names need only be unique within a single type.

### Conventions

To avoid unnecessary clutter, empty components are omitted when writing out identifier contexts. For example, the record is shorthand for an whose components are all empty.

## Lists

[Lists](https://webassembly.github.io/spec/core/syntax/conventions.html#syntax-list) are written as plain sequences, but with a restriction on the length of these sequence.

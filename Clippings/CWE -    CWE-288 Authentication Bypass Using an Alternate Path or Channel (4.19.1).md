For users who want to customize what details are displayed.<p>Description</p><div><table><tbody><tr><td>The product requires authentication, but the product has an alternate path or channel that does not require authentication.</td><td><a href="https://cwe.mitre.org/data/images/CWE-288-Diagram.png"><img src="https://cwe.mitre.org/data/images/CWE-288-Diagram.png"></a></td></tr></tbody></table></div><p>Common Consequences</p><div>This table specifies different individual consequences associated with the weakness. But this problem is not just in web apps.</td></tr></tbody></table></div><p>Applicable Platforms</p>This listing shows possible areas for which the given weakness could appear. These may be for specific named Languages, Operating Systems, Architectures, Paradigms, Technologies, or a class of such platforms. The platform is listed along with how frequently the given weakness appears for that instance.<p>Class: Not Language-Specific (Undetermined Prevalence)</p><p>Class: Not Technology-Specific (Undetermined Prevalence)</p><p>Class: Web Based (Undetermined Prevalence)</p><p>Demonstrative Examples</p><div><br><p>Register SECURE_ME is located at address 0xF00. A mirror of this register called COPY_OF_SECURE_ME is at location 0x800F00. The register SECURE_ME is protected from malicious agents and only allows access to select, while COPY_OF_SECURE_ME is not.</p><p>Access control is implemented using an allowlist (as indicated by acl_oh_allowlist). The identity of the initiator of the transaction is indicated by the one hot input, incoming_id. This is checked against the acl_oh_allowlist (which contains a list of initiators that are allowed to access the asset).</p><p>Though this example is shown in Verilog, it will apply to VHDL as well.</p><p>(informative)</p><p>Example Language: Verilog</p><div>module foo_bar(data_out, data_in, incoming_id, address, clk, rst_n);<br>output [31:0] data_out;<br>input [31:0] data_in, incoming_id, address;<br>input clk, rst_n;<br>wire write_auth, addr_auth;<br>reg [31:0] data_out, acl_oh_allowlist, q;<br>assign write_auth = | (incoming_id & acl_oh_allowlist)? 1: 0;<br>always @*<br><p>acl_oh_allowlist <= 32'h8312;<br></p>assign addr_auth = (address == 32'hF00)? 1: 0;<br>always @ (posedge clk or negedge rst_n)<br><div>if (!rst_n)<br><div>begin<br><p>q <= 32'h0;<br>data_out <= 32'h0;<br></p>end<br></div>else<br><div>begin<br><p>q <= (addr_auth & write_auth)? data_in: q;<br>data_out <= q;<br></p>end<br></div>end<br></div>endmodule<br></div><p>(bad code)</p><p>Example Language: Verilog</p><p>assign addr_auth = (address == 32'hF00)? 1: 0;</p><p>The bugged line of code is repeated in the Bad example above. Weakness arises from the fact that the SECURE_ME register can be modified by writing to the shadow register COPY_OF_SECURE_ME, the address of COPY_OF_SECURE_ME should also be included in the check. That buggy line of code should instead be replaced as shown in the Good Code Snippet below.</p><p>(good code)</p><p>Example Language: Verilog</p><p>assign addr_auth = (address == 32'hF00 || address == 32'h800F00)? 1: 0;</p><br><br></div><p>Selected Observed Examples</p><p><i>Note: this is a curated list of examples for users to understand the variety of ways in which this weakness can be introduced.
